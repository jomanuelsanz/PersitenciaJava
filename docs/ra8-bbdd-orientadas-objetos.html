1. Introducción a las bases de datos orientadas a objetos

¿Qué son las BBDD orientadas a objetos?

Una base de datos orientada a objetos (OODB) almacena información directamente como objetos, del mismo modo en que los lenguajes de programación orientados a objetos (como Java) los representan en memoria. No es necesario transformar los objetos a tablas ni filas: el propio objeto —con sus atributos y relaciones— es la unidad de persistencia.

Los principales conceptos que introduce una OODB son:

  • OID (Object Identifier): cada objeto persistente recibe un identificador único e inmutable generado por la propia base de datos.
  • Persistencia por alcance (persistence by reachability): al guardar un objeto, todos los objetos a los que referencia también se persisten en cascada.
  • Identidad vs. igualdad: dos objetos pueden tener los mismos valores pero ser objetos distintos en la base de datos.
  • Ciclo de vida: un objeto puede estar en estado transitorio (solo en memoria), persistente (guardado en la BBDD) o eliminado.

Comparativa: BBDD relacionales vs. orientadas a objetos

AspectoBBDD Relacional (SQL)BBDD Orientada a Objetos
Unidad de datosFila en una tablaObjeto Java
EstructuraTablas con columnas fijasClases con atributos y métodos
RelacionesClaves foráneas (FK)Referencias directas entre objetos
ConsultasSQLJPQL / JDOQL / consultas nativas
HerenciaEstrategias de mapeo complejasNatural (como en Java)
Impedance mismatchSí (ORM necesario)No (objetos directos)
Uso principalDatos tabulares, informesGráficos de objetos complejos
El problema del impedance mismatch es la diferencia conceptual entre el modelo de objetos de Java y el modelo relacional de SQL. Las BBDD OO lo eliminan, pero las BBDD relacionales siguen siendo más comunes en producción.

Sistemas gestores OODB más conocidos:

  • ObjectDB — Java puro, soporta JPA y JDO, activamente mantenido. El que usaremos.
  • db4o — Clásico en entornos educativos; actualmente sin mantenimiento activo.
  • Versant / Objectivity — Entornos empresariales.

2. Persistencia básica: Serialización de objetos en Java

Antes de usar un OODB completo, Java ofrece un mecanismo nativo de persistencia: la serialización. Convierte un objeto Java en una secuencia de bytes que puede guardarse en un fichero o transmitirse por red, y luego reconstruirse exactamente igual.

Es un buen punto de partida conceptual, pero no es una base de datos real: no ofrece índices, no permite consultas, y escala mal con muchos objetos.

2.1 La interfaz Serializable

Para que un objeto pueda serializarse, su clase debe implementar java.io.Serializable. Es una interfaz marcador: no define ningún método, solo indica que la clase permite la serialización. 

import java.io.Serializable;

public class Producto implements Serializable {

    private static final long serialVersionUID = 1L;

    private String nombre;
    private double precio;
    private int    stock;

    public Producto(String nombre, double precio, int stock) {
        this.nombre = nombre;
        this.precio = precio;
        this.stock  = stock;
    }

    // getters y setters ...
}

2.2 ObjectOutputStream y ObjectInputStream

La escritura se realiza con ObjectOutputStream y la lectura con ObjectInputStream.

Guardar un objeto en disco

import java.io.*;

public class GuardarProducto {
    public static void main(String[] args) {

        Producto p = new Producto("Teclado", 49.99, 10);

        try (ObjectOutputStream oos = new ObjectOutputStream(
                new FileOutputStream("producto.dat"))) {

            oos.writeObject(p);
            System.out.println("Objeto guardado correctamente.");

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

Recuperar el objeto del disco

import java.io.*;

public class RecuperarProducto {
    public static void main(String[] args) {

        try (ObjectInputStream ois = new ObjectInputStream(
                new FileInputStream("producto.dat"))) {

            Producto p = (Producto) ois.readObject();
            System.out.println("Nombre: " + p.getNombre());
            System.out.println("Precio: " + p.getPrecio());

        } catch (IOException | ClassNotFoundException e) {
            e.printStackTrace();
        }
    }
}

2.3 transient y serialVersionUID

  • transient: marca un campo para que no se serialice (contraseñas, conexiones, caché calculada).
  • serialVersionUID: número de versión de la clase. Si la clase cambia sin actualizarlo, la deserialización puede fallar con InvalidClassException. Siempre declararlo explícitamente. 
public class Usuario implements Serializable {

    private static final long serialVersionUID = 2L;

    private String nombre;
    private transient String password;  // NO se guarda
    private transient Connection conn;   // NO se guarda
}
La serialización nativa no es segura para datos no confiables. Oracle recomienda evitar deserializar datos de fuentes externas desconocidas. Ver: Java Object Serialization Specification (Java 17).

3. ObjectDB y JPA: base de datos orientada a objetos real

ObjectDB es una base de datos orientada a objetos pura en Java que implementa el estándar JPA (Jakarta Persistence API). Esto nos permite aprender JPA —que también se usa con BBDD relacionales— trabajando directamente con objetos sin ningún mapeo a tablas.

3.1 Configuración del proyecto Maven

Añade el repositorio y la dependencia de ObjectDB a tu pom.xml:

<!-- pom.xml -->
<repositories>
  <repository>
    <id>objectdb</id>
    <url>https://m2.objectdb.com</url>
  </repository>
</repositories>

<dependencies>
  <!-- ObjectDB (jakarta variant, requiere Java 17+) -->
  <dependency>
    <groupId>com.objectdb</groupId>
    <artifactId>objectdb-jk</artifactId>
    <version>2.9.5</version>
  </dependency>
</dependencies>

Crea también el fichero de configuración de persistencia en src/main/resources/META-INF/persistence.xml: 

<?xml version="1.0" encoding="UTF-8"?>
<persistence version="3.0"
    xmlns="https://jakarta.ee/xml/ns/persistence"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/persistence
        https://jakarta.ee/xml/ns/persistence/persistence_3_0.xsd">

  <persistence-unit name="tiendaDB">
    <provider>com.objectdb.jpa.Provider</provider>
    <properties>
      <!-- Fichero de base de datos (se crea automáticamente) -->
      <property name="javax.persistence.jdbc.url"
                value="tienda.odb" />
    </properties>
  </persistence-unit>

</persistence>

3.2 Entidades JPA

Una entidad es una clase Java anotada con @Entity que JPA mapeará como un tipo persistente. Requiere un campo marcado con @Id como identificador único. 

import jakarta.persistence.*;

@Entity
public class Producto {

    @Id
    @GeneratedValue  // ObjectDB genera el ID automáticamente
    private long id;

    private String nombre;
    private double precio;
    private int    stock;

    // Constructor vacío obligatorio para JPA
    public Producto() {}

    public Producto(String nombre, double precio, int stock) {
        this.nombre = nombre;
        this.precio = precio;
        this.stock  = stock;
    }

    // Getters y setters ...

    public long   getId()     { return id; }
    public String getNombre() { return nombre; }
    public void   setNombre(String n) { nombre = n; }
    public double getPrecio() { return precio; }
    public void   setPrecio(double p) { precio = p; }
    public int    getStock()  { return stock; }
    public void   setStock(int s) { stock = s; }
}
JPA exige un constructor vacío (público o protegido) para poder reconstruir los objetos al leerlos de la base de datos.

3.3 EntityManager: operaciones CRUD

El EntityManager es la interfaz principal de JPA. Se obtiene a través de un EntityManagerFactory y gestiona el ciclo de vida de los objetos persistentes.

Patrón de uso estándar

import jakarta.persistence.*;

public class Main {
    public static void main(String[] args) {

        EntityManagerFactory emf =
            Persistence.createEntityManagerFactory("tiendaDB");
        EntityManager em = emf.createEntityManager();

        try {
            // ─── CREATE ──────────────────────────────
            em.getTransaction().begin();
            Producto teclado = new Producto("Teclado", 49.99, 10);
            em.persist(teclado);          // guarda el objeto
            em.getTransaction().commit();
            System.out.println("ID asignado: " + teclado.getId());

            // ─── READ ─────────────────────────────────
            Producto encontrado = em.find(Producto.class, teclado.getId());
            System.out.println("Encontrado: " + encontrado.getNombre());

            // ─── UPDATE ───────────────────────────────
            em.getTransaction().begin();
            encontrado.setPrecio(44.99);  // JPA detecta el cambio
            em.getTransaction().commit();

            // ─── DELETE ───────────────────────────────
            em.getTransaction().begin();
            em.remove(encontrado);
            em.getTransaction().commit();

        } finally {
            em.close();
            emf.close();
        }
    }
}
OperaciónMétodo JPADescripción
Crearem.persist(obj)Inserta el objeto en la BBDD
Leer por IDem.find(Clase.class, id)Devuelve el objeto o null
ActualizarModificar campo + commit()JPA detecta cambios automáticamente
Eliminarem.remove(obj)Elimina el objeto de la BBDD
Adjuntarem.merge(obj)Sincroniza un objeto detached

4. Consultas con JPQL

JPQL (Jakarta Persistence Query Language) es el lenguaje de consulta estándar de JPA. Es similar a SQL pero opera sobre clases y atributos Java, no sobre tablas y columnas.

4.1 SELECT básico

// Recuperar todos los productos
TypedQuery<Producto> query = em.createQuery(
    "SELECT p FROM Producto p", Producto.class
);
List<Producto> todos = query.getResultList();

// Ordenar por precio descendente
List<Producto> ordenados = em.createQuery(
    "SELECT p FROM Producto p ORDER BY p.precio DESC",
    Producto.class
).getResultList();

4.2 Filtros y parámetros nombrados

// Productos con precio menor a un valor (parámetro nombrado)
TypedQuery<Producto> q = em.createQuery(
    "SELECT p FROM Producto p WHERE p.precio < :maxPrecio",
    Producto.class
);
q.setParameter("maxPrecio", 50.0);
List<Producto> baratos = q.getResultList();

// Contar registros
long total = em.createQuery(
    "SELECT COUNT(p) FROM Producto p", Long.class
).getSingleResult();

// Buscar por nombre (LIKE)
List<Producto> resultado = em.createQuery(
    "SELECT p FROM Producto p WHERE p.nombre LIKE :patron",
    Producto.class
).setParameter("patron", "%teclado%")
 .getResultList();
Usa siempre parámetros nombrados (:nombre) en lugar de concatenar cadenas en la consulta. La concatenación puede introducir vulnerabilidades de inyección.

5. Tipos estructurados, compuestos y relacionados

5.1 Herencia entre entidades

JPA soporta jerarquías de clases de forma natural. La anotación @Inheritance controla cómo se almacena la jerarquía. Con ObjectDB (OODB), la herencia funciona de manera transparente. 

@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
public abstract class Vehiculo {
    @Id @GeneratedValue
    private long id;
    private String matricula;
    private int    anio;
    // getters/setters ...
}

@Entity
public class Coche extends Vehiculo {
    private int numeroPuertas;
}

@Entity
public class Moto extends Vehiculo {
    private boolean tieneSidecar;
}
// Consultar todos los vehículos (incluye Coche y Moto)
List<Vehiculo> todos = em.createQuery(
    "SELECT v FROM Vehiculo v", Vehiculo.class
).getResultList();

// Consultar solo los coches
List<Coche> coches = em.createQuery(
    "SELECT c FROM Coche c", Coche.class
).getResultList();

5.2 Relaciones entre objetos

JPA gestiona relaciones mediante anotaciones. Los dos casos más comunes en RA8:

Relación @ManyToOne / @OneToMany

@Entity
public class Categoria {
    @Id @GeneratedValue
    private long id;
    private String nombre;

    @OneToMany(mappedBy = "categoria", cascade = CascadeType.ALL)
    private List<Producto> productos = new ArrayList<>();

    // getters/setters ...
}

@Entity
public class Producto {
    @Id @GeneratedValue
    private long id;
    private String nombre;

    @ManyToOne
    private Categoria categoria;  // referencia directa al objeto

    // getters/setters ...
}
// Crear categoría con productos y persistir en cascada
em.getTransaction().begin();

Categoria electronica = new Categoria("Electrónica");
Producto teclado = new Producto("Teclado", 49.99, 10);
teclado.setCategoria(electronica);
electronica.getProductos().add(teclado);

em.persist(electronica);  // CascadeType.ALL persiste también los productos
em.getTransaction().commit();

6. Transacciones e integridad de datos

Una transacción es un conjunto de operaciones que deben ejecutarse todas o ninguna. JPA las gestiona con la interfaz EntityTransaction.

  • begin() — inicia la transacción.
  • commit() — confirma todos los cambios en la BBDD.
  • rollback() — deshace todos los cambios desde el último begin().
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();

    // Transferencia de stock entre dos productos
    Producto origen  = em.find(Producto.class, 1L);
    Producto destino = em.find(Producto.class, 2L);

    int cantidad = 3;
    if (origen.getStock() < cantidad)
        throw new IllegalStateException("Stock insuficiente");

    origen.setStock(origen.getStock()  - cantidad);
    destino.setStock(destino.getStock() + cantidad);

    tx.commit();  // ambos cambios se guardan juntos
    System.out.println("Transferencia completada.");

} catch (Exception e) {
    if (tx.isActive()) tx.rollback();  // ningún cambio se aplica
    System.err.println("Error: " + e.getMessage());
}
El patrón try / catch / rollback garantiza que si ocurre cualquier error en mitad de la transacción, la base de datos queda en el estado anterior, sin datos a medias.
